# Competencies Bank (canonical match inventory)

This is the **single source of truth** for what the applicant can truthfully claim. The resume firewall (`resumes.md`, step 3) matches a JD's terms **against this bank**: a term that's here (and true) is swappable; a term that isn't is **asked about**, never silently dropped or added.

> **Missing ≠ false.** If a JD names a skill that isn't in this bank, do NOT silently drop it and do NOT silently add it. A missing skill is "unconfirmed." Surface it to the applicant: "JD wants X — not in the bank. Do you have an example?" If they confirm with a real instance, add it here (domain tag + honest proficiency + evidence anchor) as a deliberate, verified act — THEN it's swappable. If they can't source it, it stays out and is flagged-absent in the tailoring. **The asking step is the safeguard in both directions.**

---

## Conventions

- **Domain tags:** `[PM]` product/business · `[SW]` software/digital design · `[PH]` physical/industrial design. Many skills carry multiple.
- **Proficiency honesty:** `(core)` = proven, lead with it · `(working)` = genuinely done it and can speak to it, but not an expert; usable with care · `(learning)` = ramping, do NOT headline. Firewall rule: the bucket-2 synonym swap is allowed only for `(core)`/`(working)`, **NEVER `(learning)`.**
- **Evidence anchors** should be noted where a claim might be challenged — name the project or situation that proves it.

---

## Format

Each line follows:

```
Skill name (proficiency) [DOMAIN TAG]  · evidence anchor if challenged
```

Group entries under the same domain headings used in `projects.md` and `domains.md` for consistency. Use `·` to separate optional evidence anchor.
